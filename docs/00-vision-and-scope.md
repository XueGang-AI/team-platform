# 00 - 愿景与产品边界

> 本文明确「团队统一项目治理平台」要解决什么、不解决什么，以及各方责任边界。
> 它是所有后续架构、领域模型和路线图的出发点。如果某项设计无法回到本文找到对应的价值，则属于过度设计。

## 1. 背景

团队采用「超级个体」模式：每个成员独立负责一个完整项目，项目之间技术栈可能不同。当前存在以下治理痛点：

- 各项目缺少统一的注册、监控和治理机制；
- 日志、鉴权、配置、告警、任务、文件、模型调用等通用能力被重复开发；
- 技术负责人无法在一个统一平台查看所有项目的负责人、环境、版本、健康状态、异常、调用关系和资源消耗；
- 项目负责人离开或项目长期无人维护时，项目知识和运行信息容易丢失。

本项目解决的是**跨项目的公共治理问题**，不是某一个业务项目的具体功能。

## 2. 平台服务对象

| 对象 | 说明 |
|------|------|
| 技术负责人 / 平台管理员 | 需要全局视角，掌握所有项目的健康、异常、成本和负责人 |
| 项目负责人 | 需要自助注册项目、管理成员和环境、查看本项目可观测性 |
| 项目成员 | 在权限范围内查看和操作本项目资源 |
| 接入项目（服务/Worker/脚本） | 通过 SDK / CLI / API / OpenTelemetry 接入平台能力 |

## 3. 平台解决的问题

1. **统一服务目录**：一个地方看到所有项目、负责人、环境、版本、健康状态。
2. **统一身份与权限**：用户身份统一，项目级角色与服务身份分离，避免共享管理员凭证。
3. **统一可观测性入口**：日志、指标、链路通过统一协议接入，平台提供关联与跳转，而不是各自搭建。
4. **统一治理流程**：告警、配置、密钥、发布、任务、审计有统一模型和流程。
5. **统一接入协议**：`project.yaml` + SDK + CLI，降低接入成本，避免锁死单一技术栈。
6. **统一成本与用量治理**：模型调用、资源消耗可计量、可配额、可追溯。
7. **知识沉淀**：项目元数据、依赖关系、运行信息进入平台，降低人员变动带来的知识流失。

## 4. 平台不解决的问题

1. **不替代业务项目自身的业务逻辑实现**。平台只提供公共治理能力。
2. **不从底层重造基础设施**。不重写 Prometheus / Loki / Tempo / 对象存储 / 消息队列 / 时序数据库。
3. **不直接保存大规模原始日志、指标和 Trace 数据**。这些由外部可观测性组件承载，平台只保存元数据、索引和外部跳转链接（详见 [控制面与数据面](./01-architecture.md#4-控制面与数据面)）。
4. **不强制接入方使用某种技术栈**。平台自身用 TypeScript，接入方支持 TS / Python 等。
5. **不充当通用 PaaS / K8s 编排平台**。第一阶段不引入 Kubernetes。
6. **不托管业务项目的代码仓库与 CI/CD 流水线**。平台只记录发布事件和版本信息，与外部 Git 系统集成。

## 5. 角色定义

### 5.1 平台管理员（Platform Admin）

- 平台基础设施的维护者；
- 管理全局用户、平台级配置、集成组件（Grafana/Loki 等）的接入；
- 不属于任何具体项目，但拥有平台级只读与运维权限；
- 高风险操作必须经过审计。

### 5.2 项目负责人（Project Owner）

- 一个项目的最终责任人；
- 负责项目注册、成员管理、环境与密钥配置、发布记录维护；
- 对本项目的健康、成本、合规负责；
- 项目负责人变更必须留痕（审计事件）。

### 5.3 项目成员（Project Member）

- 在项目负责人授权范围内操作本项目资源；
- 角色粒度（maintainer / developer / viewer）由项目级 RBAC 决定；
- 不能越权访问其他项目。

### 5.4 服务身份（Service Identity）

- 接入项目以「服务」身份调用平台 API 或被平台查询；
- 服务身份与用户身份分离，使用独立的 ServiceCredential；
- 凭证可轮换、可吊销，禁止不同项目共享凭证。

## 6. 责任边界

### 6.1 平台与基础设施的责任边界

| 能力 | 责任方 |
|------|--------|
| 平台控制面（注册、权限、配置元数据、审计） | 平台自研 |
| 日志/指标/Trace 存储 | 外部组件（Loki/Prometheus/Tempo），平台集成 |
| 可视化 Dashboard | Grafana，平台提供跳转与关联 |
| 对象存储 | 外部 S3 兼容存储，平台对接 |
| 消息队列 | 外部组件，平台对接 |
| 密钥真实存储 | 外部 Secret Store / 云 KMS，平台只存元数据 |
| 时序数据 | 外部时序数据库，平台不重造 |

### 6.2 平台与业务项目的责任边界

- **平台负责**：提供注册、鉴权、可观测性接入协议、配置/密钥元数据、告警规则、发布记录、用量统计等治理能力。
- **业务项目负责**：自身业务逻辑、自身数据、自身部署（平台只记录部署事件）。
- **接入失败不互相拖垮**：平台故障不能阻断业务；SDK 必须具备超时、熔断、降级、异步上报、本地缓冲、丢弃策略（详见 [安全原则](./06-security-principles.md)）。

### 6.3 接入项目的责任边界

- 接入项目负责填写并维护 `project.yaml`、申请服务凭证、按协议上报可观测性数据；
- 平台负责校验、存储元数据、关联数据、提供查询与治理；
- 接入项目对自身上报数据的真实性负责，平台对存储和关联的完整性负责。

## 7. 当前状态

- **当前阶段**：Phase 0（仓库初始化、需求梳理、总体架构与路线规划）。
- **已实现**：仅文档与仓库基础配置，**无可运行的平台服务**。
- **未实现**：工程骨架、API、数据库、管理后台、SDK、CLI、可观测性组件部署等一切 Phase 1 及之后的内容。
