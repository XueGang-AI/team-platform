# ADR-0002 控制面与数据面分离

- **状态**：accepted
- **日期**：2026-06-17

## 背景

平台需要处理两类数据：

1. **治理元数据**：项目、成员、权限、配置、告警规则、发布记录、审计——量小、需强一致、可审计；
2. **大规模运行数据**：原始日志、指标时序、Trace span、大文件——量大、高频、时序型。

若将两者混在同一数据库，控制面会被高频数据冲垮，且治理元数据的强一致查询会被拖慢。

## 决策

**控制面与数据面分离**：

- **控制面**（PostgreSQL）：保存治理元数据、关系、规则、索引、审计，强一致可查询；
- **数据面**（外部组件）：Loki 存日志、Prometheus 存指标、Tempo 存 trace、对象存储存大文件；
- **平台不复制全量数据面数据**，只存 `project_id`/`environment`/`service_name`/`version` 等维度的关联索引与外部跳转链接。

## 理由

- **职责清晰**：控制面专注治理，数据面专注高吞吐存储；
- **避免重造**：数据面用成熟组件，不自建时序库/日志库；
- **成本可控**：高频数据由专门组件承载，控制面库不膨胀；
- **故障隔离**：数据面组件故障不影响控制面治理操作；
- **可演进**：数据面组件可独立替换。

替代方案「控制面直接存全量日志/指标」不选：成本与性能不可接受，且重造存储无差异化价值。

## 影响

- 查询可观测性数据需通过跳转或代理，平台不直接提供全量查询；
- 必须保证维度标签在采集时由 SDK/OTel 一致注入，否则无法关联；
- 数据面组件的运维成为平台依赖；
- 权限隔离需延伸到数据面跳转链接（带权限上下文）。
