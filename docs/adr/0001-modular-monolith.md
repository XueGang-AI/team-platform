# ADR-0001 采用模块化单体

- **状态**：accepted
- **日期**：2026-06-17

## 背景

平台需要承载项目注册、身份权限、可观测性适配、告警、配置、发布、任务、文件、通知、模型网关等 14+ 模块。第一阶段面临架构形态选择：直接拆成大量微服务，还是先做模块化单体。

团队规模为「超级个体」模式，运维能力有限。过早微服务化会带来服务发现、跨服务事务、分布式调试、部署复杂度等成本，远超当前收益。

## 决策

第一阶段采用**模块化单体（Modular Monolith）**：

- 单一可部署单元承载所有模块；
- 按业务域划分 NestJS Module；
- 模块间通过接口依赖注入通信，禁止跨模块直接操作对方数据库表；
- 跨模块通信走明确接口契约。

## 理由

- **运维成本低**：单一部署单元，Docker Compose 即可，无需服务发现与服务间网络治理；
- **事务简单**：跨模块一致性可在单库内处理；
- **保留可拆性**：模块边界清晰、接口契约先行，未来有真实需要时可按模块拆分为独立服务；
- **匹配团队规模**：超级个体模式下，分布式系统的运维成本不划算。

替代方案「直接微服务」当前不选：在模块边界尚未被真实流量验证前拆分，边界划错代价更高；且当前无独立伸缩需求。

## 影响

- 必须坚持模块边界纪律，避免模块间直接耦合，否则未来无法拆分；
- 单元内共享数据库，但表归属明确属于某模块；
- 未来拆分时，需先稳定接口契约再剥离数据库；
- 单体部署意味着整体发布，需关注发布粒度（通过功能开关缓解）。
